PROMPT 1 [CRITICAL] — Fix DiaryThreadScreen: _entries is always empty
────────────────────────────────────────────────────────────────────────────────

PROBLEM:
In lib/frontend/screens/diary_thread/diary_thread_screen.dart, the entries
list is declared as a top-level module-level variable:

  final _entries = <_Entry>[];

This list is never populated from DiaryStore. Methods like _togglePlay(int idx)
access _entries[idx], but the list is always empty at runtime, meaning the
diary thread renders nothing and will throw a RangeError on any play tap.

FIX:
Replace the top-level _entries declaration with an instance variable on
_DiaryThreadScreenState. Populate it inside a ListenableBuilder or override
didChangeDependencies to map DiaryStore.instance.entriesFor(widget.diaryId)
into _Entry objects. The _Entry model already has all needed fields. Map each
DiaryEntry (from diary_store.dart) to _Entry using its id, type, createdAt,
transcript, path, durationSeconds, and whether entry.fromMe is true.
Re-render the list whenever DiaryStore notifies. Remove the top-level
final _entries declaration entirely.

PROMPT 2 [CRITICAL] — Persist Me screen toggle settings to SharedPreferences
────────────────────────────────────────────────────────────────────────────────

PROBLEM:
In lib/frontend/screens/me/me_screen.dart, _notifOn and _onThisDay are plain
local bool variables initialised to true. Every app restart resets them to
true regardless of what the user toggled. These are real user preferences that
must survive restarts.

FIX:
Add a _loadPrefs() async method in _MeScreenState.initState() that reads
SharedPreferences keys 'pref_notif_on' and 'pref_on_this_day_on' and sets
state. Wrap each toggle in a _savePrefs() call that writes back to
SharedPreferences when the user flips the switch. Also wire _notifOn to
NotificationService.instance.setEnabled(bool) and _onThisDay to
OnThisDayService.instance.setEnabled(bool) so the preference actually
takes effect across the app.

PROMPT 3 [CRITICAL] — Memory Tree filter chips don't filter the MonthDetailSheet
────────────────────────────────────────────────────────────────────────────────

PROBLEM:
In lib/frontend/screens/memory_tree/memory_tree_screen.dart, the _FilterRow
sets _filter (_MType.voice or _MType.video) in _MemoryTreeScreenState, but
_filter is never passed to _TreeCanvas or _MonthDetailSheet. When a node is
tapped and the month sheet opens, it shows all entries regardless of the active
filter. Users can select the Voice filter and still see video entries in the
sheet — the filter has no effect.

FIX:
Pass _filter down to _MonthDetailSheet as a nullable _MType? parameter. Inside
_MonthDetailSheetState._entries(), add a filter step after collecting entries:
if filter == _MType.voice, keep only entries where entry.type == 'voice'.
If _MType.video, keep only 'video'. Also pass _filter to _TreeCanvas so the
_TreePainter can dim or hide node types that don't match the active filter
(reduce non-matching node alpha to 0.25). Add a 'Photo' chip to _FilterRow
since 'photo' is a valid node type in _drawNodes().

PROMPT 4 [CRITICAL] — _updateMonthKeys() called inside build() is O(n²) per frame
────────────────────────────────────────────────────────────────────────────────

PROBLEM:
In _TreeCanvasState, _updateMonthKeys() iterates all diaries and all entries
on every build call (including every animation frame from the 6-second breathe
controller). This is O(diaries × entries) per 60fps animation frame and will
cause jank as the entry count grows.

FIX:
Move _updateMonthKeys() out of build(). Call it once in initState() and again
via a DiaryStore.instance listener (addListener in initState, removeListener
in dispose). Cache the results as _monthKeys, _reactionNodeIndices, and
_monthMoodEnergies on the state object. The AnimatedBuilder that drives the
CustomPainter will still repaint on breathe ticks, but the expensive data
processing will only run when the store actually changes. Wrap the CustomPaint
builder to only pass the cached values.

Also ensure the Pulse hold gesture has a Semantics.onTap for tap-mode users

PROMPT 36 [HIGH IMPACT] — Verify ember color contrast ratios on text elements
────────────────────────────────────────────────────────────────────────────────

PROBLEM:
AppColors has a dedicated 'emberAccessible' (#FFA040) token with the comment
"WCAG AA-safe amber for small text on dark backgrounds (≥4.5:1 on inkDeep)".
But AppColors.emberBright (#FFB870) and AppColors.emberWarm (#FF9A40) are used
for small text in multiple places (eyebrow labels, filter chip labels, pulse
strip label) where contrast may fall below 4.5:1 against AppColors.ink.

FIX:
Audit every text element using ember, emberWarm, or emberBright as color:
  1. AppTypography.eyebrow() calls with ember/emberBright color
  2. FilterChip label text with widget.color (AppColors.emberWarm)
  3. Pulse strip "PULSE" eyebrow (emberBright)
  4. _StatItem value text (emberWarm)

For any element at or below 14pt that uses a non-accessible amber:
  Replace with AppColors.emberAccessible.
For elements at 16pt+ (bold) or 18pt+ (regular), emberWarm is acceptable per
WCAG AA large text rules (3:1 ratio).

Add a comment in AppColors explaining which tokens pass AA for what sizes.
