C-1  SharedPreferences.getInstance() called repeatedly — no cache
     Files:  weekly_digest_service.dart (4 separate calls per session)
             morning_service.dart (every public method)
             on_this_day_service.dart (every public method)
             notification_service.dart (_loadPrefs + setEnabled)
             user_store.dart (loadPrefs + setParentMode)
     Problem: SharedPreferences.getInstance() is an async factory that checks
              the cache itself, but still goes through a Future resolution on
              every call. Calling it 10–20× per session inside hot paths adds
              up; on low-end Android it can stall the UI thread briefly.
     Fix:    Add a static cached field to each service/store and initialise
             once in the first call or at app start:

               static SharedPreferences? _prefs;
               static Future<SharedPreferences> _getPrefs() async {
                 return _prefs ??= await SharedPreferences.getInstance();
               }

             Better: use a single app-level init in main.dart and pass or
             inject the instance so no service has to await it mid-flow:

               // main.dart
               final prefs = await SharedPreferences.getInstance();
               runApp(SaanjhApp(prefs: prefs));

C-2  setState() inside animation tick listener — morning_overlay.dart
     Lines:  ~76 (_onHoldTick callback)
     Problem: _onHoldTick() is called on every frame of _holdCtrl (up to 60×/s).
              It calls setState(() {}) which schedules a full widget rebuild of
              MorningOverlay. This is the same pattern that caused the O(n²)
              audit finding in MemoryTree.
     Fix:    Remove setState from _onHoldTick. Wrap only the widgets that
             actually depend on _holdCtrl.value in AnimatedBuilder so they
             rebuild in isolation without triggering the whole overlay rebuild:

               AnimatedBuilder(
                 animation: _holdCtrl,
                 builder: (_, __) => _HoldRing(progress: _holdCtrl.value),
               )

C-3  weekly_digest_service.dart — FlutterLocalNotificationsPlugin + timezone
     database initialised in init() which is called from HomeScreen.initState()
     Lines:  56–69
     Problem: Loading the full timezone database (timezone package) is a
              synchronous-looking but internally heavy I/O operation. Doing it
              during app startup (initState) can delay first frame.
     Fix:    Move the init() call to a postFrameCallback so the first frame
             renders before timezone data loads:

             (Already using postFrameCallback for permission ask; pull the
             init() call into the same callback.)

H-2  _grouped() in memory_jar_screen recomputes on every build
     File:   memory_jar_screen.dart  ~line 43–56
     Problem: _grouped() iterates all diaries and all their entries on every
              build() call. If DiaryStore has many entries, this is O(diaries ×
              entries) on every repaint — including repaints triggered by
              unrelated parent rebuilds.
     Fix:    Cache the grouped result and invalidate only when DiaryStore
             notifies:

               void initState() {
                 super.initState();
                 DiaryStore.instance.addListener(_onStoreChange);
                 _cachedGroups = _grouped();
               }

               void _onStoreChange() =>
                 setState(() => _cachedGroups = _grouped());

               // In build(): use _cachedGroups instead of calling _grouped()

H-3  Missing RepaintBoundary on animated CustomPainter widgets
     Files:  record_screen.dart — _Waveform widget (AnimatedBuilder + CustomPaint)
             memory_jar_screen.dart — _JarPainter (AnimatedBuilder + CustomPaint)
             on_this_day_screen.dart — _ReflectionTile play button animation
     Problem: Without RepaintBoundary, when the AnimatedBuilder ticks it can
              mark ancestor render objects dirty, causing larger subtrees to
              repaint than necessary.
     Fix:    Wrap the CustomPaint (not the AnimatedBuilder) in RepaintBoundary:

                 ),
               )

M-4  AppTypography methods create new TextStyle objects on every call
     File:   app_typography.dart (used everywhere)
     Problem: Every call to AppTypography.body(), .title(), .label() etc. calls
              GoogleFonts.inter() / GoogleFonts.cormorantGaramond() which
              constructs a new TextStyle object. In build() methods called
              60× per second (AnimatedBuilders), this generates GC pressure.
     Fix:    Cache the base TextStyles at class level and use copyWith() for
             variants. Only the base weights/sizes that appear most often need
             caching:

               // In app_typography.dart
               static final _bodyBase = GoogleFonts.inter(
                 fontSize: 15, fontWeight: FontWeight.w400,
                 letterSpacing: -0.015, height: 1.4,
               );

               static TextStyle body({double size = 15, ...}) {
                 if (size == 15 && weight == FontWeight.w400 && color == null)
                   return _bodyBase;
                 return _bodyBase.copyWith(fontSize: size, ...);
               }

             Priority: cache body, label, caption — used most in list views.
